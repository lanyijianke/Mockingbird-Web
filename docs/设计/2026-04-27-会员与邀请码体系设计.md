# 会员与邀请码体系设计

## 1. 背景

当前站点已经具备注册、登录、会员兑换、学社访问控制的基础能力，但会员角色存在一处明显歧义：

- `member` 与 `junior_member` 同时存在
- 两者当前都被视为最低会员等级
- 两者当前都可进入学社
- 邀请码系统默认发放的是 `junior_member`

这会带来三个直接问题：

1. 产品语义不清晰。运营、开发、测试很难判断“普通会员”到底对应哪个角色。
2. 权限判断容易分叉。前端 UI、API 守卫、邀请码升级规则都需要同时兼容两套最低会员角色。
3. 数据维护成本高。后续再做学社入口、会员统计、邀请码后台时，必须持续处理别名问题。

本设计的核心目标是移除 `member`，统一最低会员层级为 `junior_member`，并将会员体系与邀请码体系整理成一套可落地、可迁移、可测试的规范。

## 2. 当前实现结论

基于当前代码，现状如下：

- 新注册用户默认角色是 `user`
- 注册成功后不会自动登录
- 密码登录要求邮箱已验证
- 学社访问目前由 `/academy` 的布局层做前端门禁
- API 权限依赖 `RequireRole` + `getEffectiveRole`
- 会员过期后，`getEffectiveRole` 会把其对外视为 `user`
- 邀请码表 `InvitationCodes.TargetRole` 默认值已经是 `junior_member`

这说明：

- “刚注册完成不可进入学社”与当前实现一致
- `member` 不是“普通用户”，而是一个历史遗留的最低会员角色别名
- 移除 `member` 不会改变普通注册用户的行为，只会规范会员层级

## 3. 设计目标

本次设计要达成以下目标：

1. 统一会员角色模型，只保留一套最低会员等级。
2. 明确学社访问、会员过期、邀请码升级的语义。
3. 让前端页面、API 路由、数据库数据、测试用例使用同一套角色定义。
4. 为后续“学社入口始终可见，但仅会员可进入”的导航需求提供稳定基础。

## 4. 非目标

本设计不处理以下事项：

- 不重新定义会员价格、权益文案、支付流程
- 不新增新的会员等级
- 不重做邀请码表结构
- 不在本设计阶段直接改动业务代码

## 5. 目标角色体系

### 5.1 规范角色

移除 `member` 后，系统的规范角色集合如下：

| 角色 | 含义 | 是否可进入学社 | 是否受会员有效期约束 |
| --- | --- | --- | --- |
| `user` | 普通注册用户 | 否 | 否 |
| `junior_member` | 最低会员等级，替代原 `member` | 是 | 是 |
| `senior_member` | 高级会员 | 是 | 是 |
| `founder_member` | 创始会员 | 是 | 是 |
| `admin` | 管理员 | 是 | 否 |

### 5.2 角色层级

目标层级如下：

- `user` = 0
- `junior_member` = 1
- `senior_member` = 2
- `founder_member` = 3
- `admin` = 4

说明：

- `junior_member` 是唯一的最低会员等级
- `admin` 是系统角色，不属于邀请码升级链条中的目标角色
- `member` 不再出现在任何规范数据、常量、文案、测试基线中

### 5.3 角色命名规则

对内统一使用英文角色值：

- `user`
- `junior_member`
- `senior_member`
- `founder_member`
- `admin`

对外展示文案建议统一为：

- `user` -> 普通用户
- `junior_member` -> 普通会员
- `senior_member` -> 高级会员
- `founder_member` -> 创始会员
- `admin` -> 管理员

这里建议把当前“初级会员”的展示文案改成“普通会员”，避免和你口头定义冲突。

## 6. 学社访问规则

### 6.1 访问口径

学社允许进入的角色为：

- `junior_member`
- `senior_member`
- `founder_member`
- `admin`

禁止进入的角色为：

- `user`

### 6.2 导航与页面分流

“学社”按钮应始终可见，但最终访问结果由 `/academy` 统一裁决：

- 未登录用户：跳转 `/login?callbackUrl=/academy`
- 已登录但角色为 `user`：跳转 `/membership`
- 已登录且角色为 `junior_member` / `senior_member` / `founder_member` / `admin`：放行进入 `/academy`

这样可以保证：

- 顶栏入口对所有用户可见
- 权限真相只保留在学社路由侧
- 不需要把复杂门禁判断复制到多个按钮组件中

### 6.3 会员过期语义

会员有效期通过 `MembershipExpiresAt` 表达。

规则如下：

- `admin` 永远不受会员有效期限制
- `junior_member` / `senior_member` / `founder_member` 若过期，则对外有效角色视为 `user`
- 角色过期的判断应统一由 `getEffectiveRole` 负责
- `RequireRole`、`/api/auth/me`、学社前端门禁都必须基于“有效角色”而不是原始数据库角色判断

这意味着一个过期用户在数据库里可能仍存着原会员角色，但对外权限表现必须等同 `user`。

## 7. 邀请码体系设计

### 7.1 邀请码目标角色

邀请码 `TargetRole` 只允许以下三种：

- `junior_member`
- `senior_member`
- `founder_member`

明确禁止：

- `member`
- `admin`
- `user`

### 7.2 邀请码默认时长

在延续现有设计的前提下：

- `junior_member` 默认 30 天
- `senior_member` 默认 365 天
- `founder_member` 默认 999 * 365 天

`MembershipDurationDays` 仍然保留为邀请码可覆盖字段：

- 有明确值时，以邀请码记录为准
- 没有明确值时，回退到角色默认时长

### 7.3 邀请码升级规则

邀请码只能把用户升级到更高等级，不能平移、降级或重复覆盖。

升级规则如下：

- `user` 可升级到 `junior_member` / `senior_member` / `founder_member`
- `junior_member` 可升级到 `senior_member` / `founder_member`
- `senior_member` 可升级到 `founder_member`
- `founder_member` 不可再通过邀请码升级
- `admin` 不参与邀请码升级链条

### 7.4 过期用户与邀请码

过期会员应视为 `user` 参与升级判断。

例如：

- 过期的 `junior_member` 可重新兑换 `junior_member` 邀请码
- 过期的 `junior_member` 也可直接兑换 `senior_member` 邀请码

因此，邀请码兑换接口必须使用“有效角色”而非原始存储角色参与 `canUpgradeRole` 判定。

### 7.5 兑换成功后的写入规则

兑换成功后，事务内必须同时完成：

1. `InvitationCodes.UsedCount + 1`
2. 写入 `InvitationRedemptions`
3. 更新 `Users.Role`
4. 更新 `Users.MembershipExpiresAt`

成功后的目标行为：

- `Users.Role` 写入邀请码的 `TargetRole`
- `Users.MembershipExpiresAt` 写入新的会员截止时间

## 8. 用户生命周期

### 8.1 注册与登录

用户路径应为：

1. 用户注册
2. 创建账号，角色写入 `user`
3. 发送邮箱验证邮件
4. 用户验证邮箱
5. 用户登录
6. 用户仍为 `user`
7. 仅在兑换邀请码后升级为会员

### 8.2 会员升级路径

规范路径如下：

- `user` -> `junior_member`
- `user` -> `senior_member`
- `user` -> `founder_member`
- `junior_member` -> `senior_member`
- `junior_member` -> `founder_member`
- `senior_member` -> `founder_member`

### 8.3 会员过期路径

过期后表现为：

- 原始角色可保留
- 有效角色回退为 `user`
- 学社不可进入
- 会员兑换入口重新可见

## 9. 数据迁移设计

### 9.1 迁移目标

所有存量 `member` 数据必须归一化为 `junior_member`。

目标是不再让任何新旧代码依赖 `member` 作为长期存在的合法角色。

### 9.2 需要迁移的数据

至少包括两类：

1. `Users.Role = 'member'`
2. `InvitationCodes.TargetRole = 'member'`

### 9.3 建议迁移步骤

建议按以下顺序执行：

1. 先发布兼容代码，允许读到 `member` 时临时按 `junior_member` 解释
2. 执行数据回填
3. 验证线上已无 `member` 存量
4. 删除代码中的 `member` 常量、文案、测试基线和兼容分支

兼容窗口结束后，系统内只允许出现规范角色集合。

### 9.4 回填 SQL

建议的回填逻辑如下：

```sql
UPDATE Users
SET Role = 'junior_member'
WHERE Role = 'member';

UPDATE InvitationCodes
SET TargetRole = 'junior_member'
WHERE TargetRole = 'member';
```

### 9.5 数据约束要求

迁移完成后应满足以下约束：

- `Users.Role` 不再出现 `member`
- `InvitationCodes.TargetRole` 不再出现 `member`
- 新生成的邀请码默认目标角色仍为 `junior_member`

## 10. 代码收口建议

### 10.1 单一真相源

角色常量、角色排序、会员角色集合、学社允许角色集合、默认会员时长，应统一收口在同一套角色定义模块中。

建议保留以 `lib/auth/roles.ts` 为中心的设计，不再在多个页面文件里各自维护一套 `Set([...])`。

### 10.2 需要统一的代码面

后续实施时，应重点清理以下几类位置：

- 角色排序与升级规则
- 学社访问白名单
- 会员过期判断
- 个人中心角色文案
- 导航中“进入学社”与“加入会员”的显示逻辑
- 会员兑换页的权限判断
- 学社 API 路由守卫
- 单元测试中的种子角色

### 10.3 页面与 API 的职责划分

建议保持如下边界：

- `getEffectiveRole`：处理过期会员到 `user` 的降级语义
- `RequireRole`：处理 API 权限守卫
- `/api/auth/me`：输出前端可直接消费的有效角色
- `/academy` 布局：负责未登录、无权限、已授权三种前端分流
- 顶栏“学社”按钮：只负责提供入口，不负责复制整套权限规则

## 11. 测试设计

### 11.1 角色语义测试

至少覆盖：

- 注册后用户角色为 `user`
- 过期会员在 `/api/auth/me` 中被视为 `user`
- `junior_member` / `senior_member` / `founder_member` / `admin` 可进入学社
- `user` 不可进入学社

### 11.2 邀请码升级测试

至少覆盖：

- `user` -> `junior_member`
- `junior_member` -> `senior_member`
- `senior_member` -> `founder_member`
- 同级或降级邀请码被拒绝
- 过期会员按 `user` 重新参与升级
- 邀请码默认时长回退正确

### 11.3 迁移测试

至少覆盖：

- 旧数据中的 `member` 被正确回填为 `junior_member`
- 旧邀请码中的 `member` 目标角色被正确回填为 `junior_member`
- 回填后权限判断与兑换规则不发生回归

## 12. 风险与回滚

### 12.1 主要风险

最大风险是直接删掉 `member`，但没有先迁移存量数据，导致历史会员失去学社访问权。

### 12.2 风险控制

因此建议：

- 先兼容，再回填，再彻底删除
- 回填前先统计 `member` 存量
- 回填后复查 `Users` 与 `InvitationCodes` 中是否还存在 `member`

### 12.3 回滚原则

如果回填后发现异常：

- 优先恢复兼容分支
- 保证 `member` 临时仍按 `junior_member` 解释
- 不应回滚已成功写入的会员有效期数据

## 13. 最终决策

本设计的最终结论如下：

1. `member` 被正式废弃。
2. 最低会员等级统一为 `junior_member`，对外展示为“普通会员”。
3. 普通注册用户始终是 `user`，没有学社访问权限。
4. 学社可进入角色统一为 `junior_member`、`senior_member`、`founder_member`、`admin`。
5. 邀请码目标角色统一为 `junior_member`、`senior_member`、`founder_member`。
6. 会员过期后的有效角色统一回退为 `user`。
7. 所有 `member` 存量数据都必须迁移到 `junior_member`。

这套设计完成后，会员体系、邀请码体系、学社门禁和导航入口将共享同一套角色语义，不再存在“`member` 和 `junior_member` 到底谁是普通会员”的冲突。
